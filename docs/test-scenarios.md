# Сценарии тестирования

Документ описывает тесты, которые использовались для сравнения трех конфигураций приложения:

1. `baseline` - приложение без обработчика `SIGTERM` и без учета операций;
2. `server-shutdown` - приложение с обработчиком `SIGTERM` и вызовом `http.Server.Shutdown`, но без readiness/draining и без прикладного журнала ресурсов;
3. `module` - приложение с разработанным модулем корректного завершения.

## Методика эксперимента

Методика построена как сравнительный эксперимент. Для всех трех конфигураций использовались одинаковые параметры нагрузки, одинаковый момент инициирования остановки пода и одинаковое значение `SHUTDOWN_TIMEOUT_SECONDS`.

В каждом запуске отправляется 50 HTTP-запросов методом `GET` к маршруту `/work`. Запросы стартуют с интервалом 100 мс, но каждый запрос выполняется в отдельной goroutine, поэтому нагрузка является параллельной. Остановка пода инициируется через 3 секунды после начала нагрузки. К этому моменту часть коротких запросов уже завершена, часть средних и длительных запросов еще находится в обработке, а оставшиеся запросы продолжают стартовать.

## Контролируемое случайное распределение

Нагрузочный клиент использует контролируемое случайное распределение. Фиксированными являются доли типов запросов и границы диапазонов, а конкретная задержка каждого запроса выбирается случайно внутри диапазона:

- 70 % коротких запросов: 100-300 мс;
- 20 % средних запросов: 2-5 с;
- 10 % длительных запросов: 12-18 с;
- отдельный негативный timeout-сценарий: 1 запрос с задержкой 60 с при `SHUTDOWN_TIMEOUT_SECONDS=25`.

Для значения по умолчанию `--count 50` это дает 35 коротких, 10 средних и 5 длительных запросов. Такой подход позволяет избежать полностью одинаковых прогонов, но сохранить воспроизводимость за счет параметра `--seed`.

## Почему 50 запусков

50 повторов выбраны как компромисс между устойчивостью результата и временем выполнения эксперимента. При 10-20 запусках отдельные длительные запросы и случайный порядок их старта могут заметно смещать средние значения. 50 запусков сглаживают случайные отклонения, но остаются выполнимыми на локальном стенде Minikube. Задача эксперимента состоит не в промышленном нагрузочном тестировании, а в сравнении трех вариантов завершения приложения при одинаковых условиях.

## Запуск нагрузки

Случайная смешанная нагрузка:

```bash
go run ./cmd/load-client --target http://localhost:8080/work --interval 100ms --mode mixed --count 50
```

Воспроизводимый запуск с фиксированным seed:

```bash
go run ./cmd/load-client --target http://localhost:8080/work --interval 100ms --mode mixed --count 50 --seed 42
```

Негативный сценарий с превышением timeout:

```bash
go run ./cmd/load-client --target http://localhost:8080/work --mode timeout --timeout-delay 60s
```

Для Kubernetes можно выполнить port-forward:

```bash
kubectl port-forward svc/graceful-app 8080:80
```

## Основные сценарии

| Сценарий | Команда | Ожидаемый результат | Где отражен результат в ВКР |
|---|---|---|---|
| Удаление пода | `kubectl delete pod <pod>` | Зарегистрированные операции завершаются в пределах timeout | Таблица 3.5 |
| Rolling update, 2 реплики | `kubectl rollout restart deploy/graceful-app` | Новая нагрузка уходит на готовую реплику | Таблица 3.6 |
| Rolling update, 1 реплика | `kubectl scale deploy/graceful-app --replicas=1` и restart | Уже начатые операции сохраняются, непрерывность новых запросов не гарантируется | Таблица 3.6 |
| Scale down | `kubectl scale deploy/graceful-app --replicas=1` | Завершаемый под переходит в `draining` | Таблица 3.6 |
| Timeout | `GET /work?delay=60s&op=long-01` | В журнале появляется `shutdown_timeout` с описанием операции | Таблица 3.7, листинг 3.12 |
| Асинхронная задача | `GET /work?delay=10s&op=async-01&async=true` | В журнале появляются `async_started`, `async_done` или `shutdown_timeout` | Таблица 3.6, листинг 3.14 |
| Закрытие ресурсов | Остановка пода с модулем | В журнале фиксируются `resource_close` для `demo_db` и `demo_queue_client` | Таблица 3.6, листинг 3.13 |

## Примеры журналов

Успешное завершение:

```json
{"event":"shutdown_requested","source":"preStop","active_ops":4}
{"event":"readiness_changed","state":"draining","active_ops":4}
{"event":"active_ops_wait","active_ops":2}
{"event":"active_ops_done","active_ops":0,"result":"all_registered_operations_completed"}
{"event":"resource_close","resource":"demo_db","result":"ok"}
{"event":"resource_close","resource":"demo_queue_client","result":"ok"}
{"event":"shutdown_complete","result":"graceful"}
```

Превышение timeout:

```json
{"event":"shutdown_timeout","active_ops":1,"timeout_ms":25000,"operations":[{"op_id":"long-01","method":"GET","path":"/work","remote_ip":"10.244.0.x","elapsed_ms":25000}],"result":"forced_by_app_policy"}
```
